Synergy ≠ Similarity — CRITICAL RULE

PREFER combinations where repos occupy DIFFERENT layers of a system or provide truly complementary capabilities.

STRONGLY PENALIZE pairing repos that are direct substitutes (same role in the stack, same problem domain, overlapping integration surfaces, near-identical keywords) UNLESS the collaboration concept is explicitly one of these narrow exceptions:

  - Migration tooling: tooling to move data or configs between the two systems
  - Interoperability adapter: unified API or protocol bridge between the two
  - Benchmark suite: side-by-side evaluation of competing approaches
  - Shared standard/spec: both implement or consume the same protocol, schema, or format

If two repos solve the same problem in the same way, they are competitors, not collaborators. Do NOT fabricate a concept just to use them together.

Good synergy examples (different layers):
  - A vector database (storage layer) + an embedding model server (inference layer)
  - A data pipeline framework (orchestration) + a schema registry (schema layer)
  - A query engine (consumer) + an object storage backend (producer)

Bad pairings to reject:
  - Two vector databases (same role, direct competitors)
  - Two embedding frameworks with identical surfaces (duplicates)
  - Two monitoring tools with the same metrics format (redundant)

Competitor Detection — APPLY BEFORE WRITING

Before writing the brief, compare repo pairs:

1. If two repos share >50% of primary keywords AND have the same integration_surface entries → treat as competitors.
2. If their problem_summary descriptions solve the same problem for the same audience → treat as competitors.
3. Competitors may only appear in the same brief if the concept is explicitly: migration tooling, interoperability adapter, benchmark suite, or shared standard.
4. If repos share only broad ecosystem keywords (e.g., “AI”, “LLM”, “RAG”) but there is no specific integration_surface alignment, treat the pairing as weak and prefer other groupings.
5. If no valid non-competitor grouping is possible from the provided repos, output the best available grouping and note the limitation in the concept field.

Integration Role Definitions

Use one of:

producer — generates data, signals, artifacts, or services consumed by others

consumer — depends on or integrates outputs from another repo

middleware — connects, transforms, orchestrates, or bridges multiple repos

peer — complementary tool operating alongside another without strict dependency

Role correctness rules:
  - Two databases are NOT producer/consumer by default — assign peer or middleware only if a concrete data flow exists between them.
  - A consumer repo must actually depend on or call the producer repo's output; do not assign producer/consumer to repos that merely coexist.
  - middleware must sit between two other repos; do not use it for standalone tools.
  - When in doubt between producer and peer, choose peer.

Choose the most defensible role based strictly on the provided integration_surface and problem_summary data.
